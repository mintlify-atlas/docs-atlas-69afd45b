---
title: Frequently asked questions
description: Common questions about lakhua's coverage, performance, use cases, and limitations
---

<AccordionGroup>
  <Accordion title="Is lakhua online or offline?">
    Offline. The SDK loads local JSON data into memory and performs in-process lookups. No network calls are made after the initial data load.
    
    <Info>
      All reverse geocoding happens entirely within your application's process — coordinates never leave your system.
    </Info>
  </Accordion>

  <Accordion title="Does it support countries outside India?">
    No. Current datasets target India only. The library is specifically optimized for Indian geographic data using H3 indexing of OpenStreetMap data.
  </Accordion>

  <Accordion title="Why can some coordinates return no result?">
    Coverage is not exhaustive. If no matching cell exists (including fallback parent lookup), SDKs return `null`/`None`/`nil`.
    
    <Note>
      Rural or remote areas may return `null` as the dataset focuses on major cities and districts. If you encounter missing coverage, please open an issue with the coordinates.
    </Note>
  </Accordion>

  <Accordion title="What is fallback?">
    When enabled (default), lookup tries exact resolution first (typically `5`) and then parent resolution (`4`) if needed.
    
    This hierarchical approach means that even if your exact coordinate doesn't have an entry at resolution 5, it can still match a larger area at resolution 4, increasing coverage.
    
    You can disable fallback by setting `fallback: false` in options if you only want exact matches.
  </Accordion>

  <Accordion title="Is the first call slower?">
    Usually yes. First call reads data files into memory. Later calls are memory lookups and are much faster.
    
    The initial load typically takes a few milliseconds to read the JSON files (`reverse_geo_4.json` and `reverse_geo_5.json`). After that, lookups are sub-millisecond map operations.
  </Accordion>

  <Accordion title="Can I use H3 indices directly?">
    Yes. If you already have H3 indices, you can look them up directly:
    
    - **JavaScript**: `geocodeH3(h3Index)`
    - **Python**: `geocode_h3(h3_index)`
    - **Go**: `GeocodeH3(h3Index, options)`
    
    This is useful if you're already working with H3 in your application and want to skip the lat/lon to H3 conversion step.
  </Accordion>

  <Accordion title="Is this suitable for production?">
    Yes, for India-focused, low-latency reverse geocoding use cases where city/state-level results are acceptable.
    
    <Warning>
      Not suitable if you need:
      - Address-level or rooftop precision
      - Global coverage beyond India
      - Live POI or business metadata
      - Guaranteed coverage for every coordinate in India
    </Warning>
  </Accordion>

  <Accordion title="How do I debug lookup behavior?">
    Enable debug mode in options to see load and lookup timing logs:
    
    - **JavaScript**: `geocode(lat, lon, { debug: true })`
    - **Python**: `geocode(lat, lon, GeocodeOptions(debug=True))`
    - **Go**: `Geocode(lat, lon, &GeocodeOptions{Debug: true})`
    
    Debug mode will print timing information for data loading and lookup operations to help you understand performance characteristics.
  </Accordion>

  <Accordion title="What level of precision does lakhua provide?">
    lakhua provides city/state/district-level precision with optional pincode data. It does not provide street addresses or building-level precision.
    
    Best for:
    - Analytics dashboards showing user distribution by city
    - Backend services that need high-level location context
    - Batch processing of GPS coordinates for regional insights
    - Telemetry systems tracking device locations by district/state
  </Accordion>

  <Accordion title="How much memory does lakhua use?">
    The unpacked package size is approximately 1.39 MB. Once loaded into memory, the data structures occupy a similar footprint.
    
    This makes lakhua suitable for memory-constrained environments where hosted API alternatives would require managing connection pools and rate limits.
  </Accordion>

  <Accordion title="What runtimes are supported?">
    - **JavaScript SDK**: Node.js and Bun (requires local filesystem access for packaged data)
    - **Python SDK**: CPython 3.8+
    - **Go SDK**: Go 1.21+
    
    <Note>
      The JavaScript SDK requires filesystem access to read the bundled JSON data files, so it may not work in browser or edge environments without modifications.
    </Note>
  </Accordion>

  <Accordion title="How often is the data updated?">
    The dataset is based on OpenStreetMap data and is updated periodically. Check the project's GitHub repository for the latest data version and update schedule.
  </Accordion>
</AccordionGroup>
